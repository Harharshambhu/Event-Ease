Module 2: Catering — meal request, approval & check-in
Phase 1 — Configuration (admin)
ElementsTasksObjectives
Event daysLoad-in, show days, load-out Define which days include catering — include load-in and load-out days, not just show days. Set start/end dates per day Ensure meal forecasting covers the full event lifecycle, not just performance days
Meal typesCustom names, per-day availability Create meal types: Breakfast, Lunch, Dinner, Snack Bar, Smoothie Station, Boxed Meal, Grab-and-Go — unlimited custom types. Set which meals are available on which days Match the actual catering operation — not every meal runs every day, and non-standard meals (snack bars, coffee stations) need tracking too
Dietary restrictionsConfigurable options per event Set up restriction categories: vegetarian, vegan, gluten-free, halal, kosher, nut allergy — preset or custom. Recommend 4-10 options. Collect allergies as a separate field Give the catering vendor accurate dietary data so they can plan portions and prevent allergic reactions
Item blocksMeal packages per group type Create catering item blocks: which meals each group type can request, quantity limits, pricing if applicable. Assign blocks to portals Control who can request what — volunteers get standard meals, VIPs get premium dining, artists get hospitality
Approver settingsWho can approve/deny meals Assign team members with approver privileges for catering. Configure manager override code for onsite exceptions Ensure meal approvals go through authorized staff — prevent unauthorized meal orders inflating the catering budget
Phase 2 — Request & collection (stakeholders)
ElementsTasksObjectives
Group-level meal requestsVia portal or form Group contact requests meals for their entire team: "12 lunches and 12 dinners on load-in day, 15 lunches on show day." Selects meal types from available options Get aggregate meal counts per group per day for catering vendor forecasting
Person-level meal assignmentIndividual tracking Assign specific meals to specific people within a group — enables individual-level check-in rather than group-level only Know exactly who has a meal, not just "the vendor group has 12 lunches" — prevents sharing and overspend
Dietary info per personCollected via form fields Each person's dietary restrictions are captured as part of their profile — persists across modules (catering, credentials, CRM) One-time collection of dietary info that flows automatically to every relevant report and catering order
Admin manual entryFor late additions Admin manually adds meal orders for walk-ins, last-minute crew, or executive additions Handle exceptions without breaking the self-service model
Phase 3 — Approval & forecasting (admin)
ElementsTasksObjectives
Approval queuePending meal requests Review and approve/deny meal requests per group. Admin-created orders auto-approve. Portal/form submissions require manual approval Control the catering budget — every meal is an explicit cost; unauthorized meals inflate spend
Catering dashboardForecasting by day and meal View total approved/pending/denied/unused/used meals broken down by day and meal type. Export CSV/Excel for catering vendor Give the catering vendor exact numbers per meal per day — no more guessing or over-ordering
Budget trackingMeals as cost items Track total meal cost by group, by day, by meal type. Identify over-requests and under-utilization patterns Treat catering as a controllable line item, not an untracked cost that balloons every year
Phase 4 — Onsite check-in (event day)
ElementsTasksObjectives
RFID / barcode scanningWristband → meal match Scan person's wristband at catering station. System matches wristband to person record, finds their approved meals, and claims the meal Instant verification — only people with approved meals eat. No printed meal tickets to lose or forge
Group-level check-inSearch by group name If no individual wristband, search by group name and claim a meal from the group's pool Flexible fallback — handles groups that didn't assign meals to individuals
Manager overridePassword-protected exception If a wristband has no meal order, manager enters override code to approve the meal and track it against the wristband ID Handle day-of exceptions without blocking the check-in line — log it, don't deny it
Real-time usage dashboardApproved vs. used vs. unused Live dashboard shows how many meals have been checked in vs. how many remain. Broken down by day, meal type, group Spot waste in real time — if 40% of dinners go unused, adjust tomorrow's order and save budget
