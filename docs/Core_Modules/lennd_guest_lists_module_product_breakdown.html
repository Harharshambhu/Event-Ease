Module 4: Guest lists — allocation, self-management & fulfillment
Guest lists work on an allocation model — fundamentally different from the other three modules. The admin doesn't collect requests and approve them. Instead, the admin pre-sets a quota ("Artist X gets 8 guest passes") and the stakeholder fills in the names within that limit. The control is the cap, not the approval.
Phase 1 — Allocation setup (admin)
ElementsTasksObjectives
Pass type definitionsTypes available for guest lists Define which credential/pass types are available for guest list allocation: GA tickets, VIP passes, backstage, meet-and-greet, hospitality. Each type can have different access levels Differentiate between "8 general admission guest passes" and "2 VIP hospitality passes" — not all guest spots are equal
Meal allocationsCatering tied to guest list Optionally attach meal allocations to guest list spots — e.g., each VIP guest also gets a dinner meal. Meals auto-create in catering module when guest names are submitted Link guest access and catering in one allocation decision — no separate meal request needed per guest
Group-level quotasHow many spots each group gets Set allocation per group: "DJ Shadow gets 8 GA + 2 VIP." "RedBull Sponsor gets 50 GA + 10 VIP." Quotas are hard limits — the portal enforces the cap Budget control at the allocation level — the total guest list size is decided upfront, not negotiated name-by-name
Unique allocation groupsCustom segmentation Create named allocation groups beyond standard group types: "Headliner guests," "Sponsor VIP table," "Staff family," "Contest winners." Each with its own quota and pass types Handle non-standard guest list categories that don't map to vendor/artist/sponsor — every event has unique segments
Phase 2 — Name submission (stakeholders, self-service)
ElementsTasksObjectives
Portal guest list viewStakeholder sees their allocation Stakeholder logs into portal and sees: "You have 8 GA passes — 5 names submitted, 3 remaining." Clear count with visual progress Stakeholder knows exactly how many spots they have and how many remain — no guessing, no over-requesting
Name entryAdd/edit/remove guest names Stakeholder enters guest names, plus-one info, contact details. Can edit or swap names until the deadline. Portal enforces the quantity cap — cannot add name #9 if allocation is 8 Push the name-management burden to the person who actually knows the guests — the artist's manager, the sponsor's marketing lead
Deadline enforcementConfigurable close date Admin sets a submission deadline. Portal locks name entry after the deadline. Late changes require admin intervention Force timely submission so the box office has a finalized list before event day — no more last-minute chaos
Guest self-sufficiencyReducing admin involvement Stakeholder manages their own list end-to-end: add, edit, remove, view status. Admin only intervenes for deadline extensions or quota changes Eliminate the "can you add one more name?" email chain — the stakeholder does it themselves within their cap
Phase 3 — Management & reporting (admin)
ElementsTasksObjectives
Completion trackingWho submitted, who hasn't Dashboard shows per-group: total allocated, names submitted, names remaining, submission status (complete / partial / empty). Flag overdue groups Know at a glance which groups haven't submitted names — proactive follow-up instead of day-of surprises
Aggregate guest countTotal across all groups View total guest list size: sum of all allocated spots, all submitted names, all pass types. Broken down by allocation group Give security and box office accurate total headcount for staffing, credential printing, and venue capacity planning
Export for box officeCSV/Excel for will-call Export finalized guest list with names, pass types, associated group, and barcode/ticket info. Formatted for box office will-call system Seamless handoff from the portal to the onsite pickup station — no reformatting, no manual list creation
Quota adjustmentAdmin override on allocations Increase or decrease a group's allocation after initial setup. Handles "the headliner's manager needs 4 more passes" without rebuilding the whole list Flexibility within structure — accommodate changes without losing the data integrity of the allocation model
Phase 4 — Fulfillment & onsite
ElementsTasksObjectives
Credential generationPasses for submitted guests Submitted guest names flow into the credentials module. Passes/badges generated per guest with correct access level. Barcode assigned automatically Guest list names become scannable credentials — the system closes the loop from "name on a list" to "badge in hand"
Catering syncMeals for allocated guests If meal allocations were attached to guest list spots, submitted guests automatically appear in the catering module with their assigned meals One allocation decision flows into two modules — guest pass + guest dinner created from a single name submission
Will-call / pickupOnsite guest credential distribution Guest arrives at will-call, name is scanned or searched, credential is issued. Check-in app marks the guest as arrived Smooth guest arrival experience — no waiting, no "I'm not on the list" drama, no handwritten additions
No-show trackingUnused allocations Post-event: see which allocated guest spots were unused (names never submitted or guests never picked up credentials) Year-over-year optimization — if an artist consistently uses only 4 of their 8 spots, allocate 5 next year and save passes for others
