The core insight

The 63 tools aren't 63 different problems. They're the same ~12 interaction patterns applied to different data objects and contexts. The matrix inflated the count because it named each instance separately — "Credential Type Manager," "Meal Type Manager," "Dietary Restriction Builder" are all one pattern: a settings page for managing entity types.

The architectural opportunity: Build 7 Foundation primitives, configure them per module and context. This isn't abstraction for its own sake — it's how every mature B2B SaaS platform (Jira, Salesforce, Hubspot, Eventbrite) is actually built. The "tool" the user sees is a configured instance of a generic primitive, not a bespoke implementation.

Framework: Objectives → Tasks → Tools

Before rationalizing tools, it helps to clearly separate what the platform is trying to accomplish at each phase. Every tool exists to serve a task; every task serves an objective.

PHASE 01
Configure

Define the rules of the event before it begins. Admins set types, limits, templates, workflows, and permissions.

Tasks: Define entity types · Set capacity limits · Design templates · Assign approvers · Build item blocks
PHASE 02
Collect

Gather requests, rosters, and data from stakeholders. The system accepts input through multiple channels.

Tasks: Submit requests · Enter rosters · Import bulk data · Track deadlines · View allocations
PHASE 03
Process

Review, approve, and manage incoming data. Admins action requests; the system generates forecasts and assigns codes.

Tasks: Approve/reject · Monitor status · Forecast demand · Track budget · Generate assignments
PHASE 04
Operate

Execute at the event. Real-time scanning, check-in, pickup fulfillment, overrides, and post-event reporting.

Tasks: Scan credentials · Print badges · Fulfill pickups · Resolve disputes · Export reports

Key industry principles per task type

Configuration tools — the settings module pattern

Industry standard: a single settings module with a left sidebar listing entity types. Selecting a type opens a detail panel on the right. All 11 entity config tools belong here — not as separate pages, but as items in a unified settings hierarchy.

  • Sidebar: "Types" section with Credential Types, Meal Types, Dietary Options, Pass Types, Asset Catalog, etc.
  • Detail panel: list of instances + "Add new" button + inline edit on each row
  • Item Block Builder lives here as a sub-section under each relevant type
  • Reference: Shopify's product settings, Salesforce Object Manager, Hubspot deal properties
ShopifySalesforceHubspotAirtable

Approval queues — the queue + detail panel pattern

Industry standard: a filterable list on the left, detail panel on the right. Bulk actions on the list (approve all selected), single-item actions in the detail panel (approve / reject / partial / comment). Status filters: Pending / Approved / Rejected / Needs Info.

  • Partial quantity approval (Assets module) is just an additional input in the decision form — not a new tool
  • Queue items are the same data model across all modules: Requestor, Type, Status, Submitted Date, Assignee
  • Module context changes the item fields and approval actions, not the queue shell itself
  • Reference: Linear issue queue, Freshdesk ticket queue, GitHub PR review queue
LinearFreshdeskJiraGithub PRs

Onsite scan terminals — offline-first PWA pattern

Industry standard: a Progressive Web App (PWA) that stores the full guest/credential list locally on device load. Scan events write to a local queue first, sync to server when connectivity is available. This prevents the #1 failure mode at live events: network drops.

  • Mode switch: QR/Barcode scan → RFID → Manual name search (fallback order)
  • Audio + visual feedback: green flash + success sound for valid, red + error for invalid
  • Group check-in: name search → select group → confirm all → write bulk check-in
  • All three scan tools (50, 53, 54) run on the same shell with a context prop (Credentials / Meals / Assets)
  • Reference: Eventbrite Organizer, Cvent OnArrival, Brushfire
EventbriteBrushfireCventIndexedDB

Dashboards — widget canvas vs. pivot table distinction

Not all dashboards are the same interaction type. The matrix has two distinct sub-patterns that need separate implementations:

  • Widget canvas (tools 35, 40, 42, 45, 46, 56): Summary cards + bar/donut charts. Works as a configurable canvas where widgets pull from data queries. One implementation, multiple configurations.
  • Pivot / cross-tab (tool 39 — Catering Forecast): Day × Meal × Group matrix. This is a specific analytical view that requires a pivot table component, not just chart widgets. Industry standard: a spreadsheet-like pivot view with drill-down.
  • Historical analytics (tool 59 — Year-over-Year): Time-series comparison. Needs a date-range picker + comparison period selector. Closest reference: Google Analytics YoY comparison view.
MetabaseLookerGoogle AnalyticsRetool

Editable data grids — column-typed tables

The key distinction between a "form" and a "table" in this context: forms collect one record at a time; tables collect many records in parallel, with one row per entity. All 6 editable table tools use the same interaction model:

  • Column types: text, number, dropdown, date, checkbox, multi-select
  • Row actions: add row, delete row, duplicate row
  • Bulk actions: import from CSV, export to CSV, clear all
  • The Meal Request Grid (Day × Meal) is just a table where column headers are days and row headers are meal types — the same table component with locked row/column headers
  • Reference: Airtable, Notion database, Google Sheets with structured schema
AirtableNotionAG GridTanStack Table

Proposed Tool Architecture

22 total tools organized into 4 layers, plus 3 background service components. Built in dependency order — each layer depends on the layer below it. The categorization reflects build priority and coupling, not just tool type.

LAYER 01
Data Primitives
Build first. All other layers consume these.
01 — Settings Module
Left nav + detail panel. Manages all entity types, item blocks, quotas, and pricing. Every admin configuration surface lives here. Think "object manager" in Salesforce.
Replaces: 11 config tools (Credential Types, Meal Types, Dietary Options, Approver Settings, Asset Catalog, Inventory Limits, Pass Types, Event Day Planner, Group Creator, Item Block Builders ×3)
List + detail panel CRUD per entity Nested settings
02 — Data Grid
Column-typed editable table with configurable schema, inline editing, bulk import/export, and matrix mode (locked row/col headers). The single table component for all tabular data entry in the platform.
Replaces: 6 table tools (Personnel Roster, Meal Request Grid, Person-Meal Assignment, Guest Name Table, Group Quota Table, Quota Adjustment)
Column types Inline edit Matrix mode Row actions
03 — Form Engine
Field-schema-driven forms with conditional logic, multiple display modes (full-page / modal / inline), and access level control (public / portal / admin). Fields pull from entity type definitions in the Settings Module.
Replaces: 7 form tools (Count Request Form, Portal Request Form, Standalone Public Form, Custom Request Form, Dietary Field, Admin Manual Add Modal, Asset Order Page)
Field schema Conditional logic Access control Branding theme
04 — Approval Queue
Filterable list + detail panel with bulk actions and per-item decision forms. Module context determines item fields and available decision types. Partial quantity approval is a field extension on the decision form, not a separate queue.
Replaces: 3 queue tools (Credentials Queue, Catering Queue, Assets Queue + partial qty)
Status filters Bulk actions Decision form Comment thread
LAYER 02
Workflow Tools
Orchestrate data flow and process logic. Depend on Layer 01.
05 — Automation / Rule Builder
Trigger → Condition → Action configuration. Handles approval routing, reminder schedules, deadline locks, and barcode auto-assignment. Lives inside Settings Module as an "Automations" section.
Replaces: Approval Workflow Builder, Automated Reminder Rules, Deadline Banner logic
Time triggers Status triggers Action library
06 — Portal Shell
Stakeholder-facing portal with contextual panels: status timeline, allocation counter, deadline banner. Panels render based on the stakeholder's module and access level. The deadline "lock" is a UI state prop — no separate implementation needed.
Replaces: Request Status View, Portal Allocation Overview, Portal Status Visibility, Deadline Banner + Lock
Status timeline Contextual panels Lock state
07 — Bulk Import Tool
CSV upload → column mapping → validation preview → import confirmation. A distinct interaction model that doesn't fit the Form Engine or Data Grid (it maps external columns to internal fields). Reusable across all modules.
Replaces: Bulk CSV Import Modal
File upload Column mapper Validation preview
08 — Report Builder
Template-based report and export generator. Templates: delivery manifest, box office export, no-show report. Output formats: PDF, CSV, Excel. Accessible from any list or dashboard view as an export action.

Report templates Format selection Data filters
LAYER 03
Intelligence Layer
Aggregation, monitoring, and analytics. Read-only surfaces that make sense of collected data.

Widget-based analytics canvas with pre-configured layouts per module. Widget types: stat card, bar chart, donut, progress bar, gauge. Static and real-time (websocket) data modes. Each module gets a default layout; layouts are customizable.

The critical decision: Tools 01–04 (Data Primitives) are the foundation everything else sits on. If you scope these correctly — genuinely configurable, not just flexible-looking — the remaining 12 tools follow at much lower cost. The risk is building tools 01–04 as "configurable" in name but bespoke in practice. Use well-established open-source primitives: TanStack Table for the Data Grid, React Hook Form for the Form Engine, a proven queue component for Approvals. Build the domain logic on top of tested primitives, not from scratch.
