Module 4: Applications — intake, evaluation & selection pipeline
Applications differ from Forms in a critical way: Forms collect data from already-known stakeholders (a vendor you've invited). Applications are the intake gate for unknown parties trying to get in — speaker proposals, vendor applications, volunteer sign-ups, media accreditation requests. The flow is: apply → review → score → accept/reject → onboard.
Phase 1 — Application design (admin)
ElementsTasksObjectives
Application form builderPublic-facing intake form Build the application form with custom fields: company info, description, portfolio links, headshot/bio upload, topic proposal, availability, references, pricing. Uses the same field library as Forms module Structured intake replaces unstructured email submissions — every applicant provides the same info in the same format for fair comparison
Application typesDifferent pipelines per stakeholder Create separate applications per use case: Call for Speakers, Vendor Application, Volunteer Sign-up, Media Accreditation, Exhibitor Application. Each with unique fields and criteria Different evaluation criteria per role — a speaker is judged on topic relevance, a vendor on pricing and capability, a volunteer on availability
Scoring criteriaEvaluation rubric for reviewers Define grading dimensions: relevance, quality, experience, cost, availability — each with a score range. Reviewers grade against these criteria. Aggregate score ranks applicants Objective, consistent evaluation — reduce "who knows who" bias with structured scoring that multiple reviewers can contribute to
Open/close datesApplication window management Set when applications open (publish date) and close (deadline). Application form URL is inaccessible before open date and rejects submissions after close date Control the application window to align with event planning timeline — not too early (incomplete info), not too late (no time to onboard)
Public URLShareable application link Each application gets a unique public URL that can be shared on the event website, social media, email campaigns, or partner networks. No account needed to apply Maximum reach — anyone can apply from anywhere without creating a Lennd account. Lower the barrier to get the best applicant pool
Phase 2 — Submission (applicants)
ElementsTasksObjectives
Application submissionZero-account form fill Applicant clicks public URL, fills out all required fields, uploads supporting files (bios, portfolio, headshots, certifications), and submits. No login or account creation needed Frictionless intake — a potential speaker or vendor submits a complete, structured application in 5-10 minutes from any device
Confirmation receiptSubmission acknowledgment Applicant receives email confirmation of submission with a summary of what they submitted. Includes info on expected review timeline Close the loop immediately — applicant knows their submission went through and what to expect next
Status visibilityIf accepted and granted portal access Once an applicant is accepted and onboarded as a group, they can see their application status in their portal: submitted, under review, accepted, rejected Transparency for accepted applicants — they see their status transition from "applicant" to "confirmed group" within the same system
Phase 3 — Review, scoring & selection (admin team)
ElementsTasksObjectives
Application review queueAll submissions in one view View all received applications in a filterable list. Sort by date submitted, score, status, or custom fields. Open individual applications to see full responses and uploaded files Process a high volume of applications efficiently — not buried in an email inbox but in a structured review interface
Multi-reviewer scoringTeam-based evaluation Multiple team members score the same application independently. Each reviewer grades against defined criteria. System aggregates scores to produce a composite ranking Collaborative evaluation with accountability — every reviewer's score is recorded, reducing individual bias and increasing fairness
Accept / reject / waitlistDecision per application Mark applications as accepted, rejected, or waitlisted. Accepted applicants auto-create a group in the event CRM. Rejection can include a reason note The accept action is the bridge from "applicant" to "participant" — one click transitions them from the application pipeline into the operational portal system
Status notification emailsAutomated decision communication System sends acceptance or rejection emails to applicants when their status changes. Customizable email templates per application type and decision Close the communication loop without manual emailing — 200 applicants each get a personalized decision email with one batch action
CRM integration on acceptanceApplicant → group → portal When an applicant is accepted, their submission data creates a group record in the event CRM. Contact info, uploaded files, and responses pre-populate the record. Admin then grants portal access and the onboarding flow begins Seamless pipeline — the data an applicant submitted in their application becomes the foundation of their operational record. No re-entry, no data loss
Phase 4 — Reporting & analytics
ElementsTasksObjectives
Application volume metricsTotal received, accepted, rejected Dashboard showing total applications received, acceptance rate, rejection rate, pending review count, average score distribution Understand your applicant funnel — is your call for speakers attracting quality proposals? Are you over-rejecting vendors?
Reviewer activity trackingWho has reviewed what See which reviewers have completed their scoring assignments and who has pending reviews. Ensure balanced workload across the review team Accountability and speed — identify if one reviewer is holding up the pipeline and redistribute work
Export for comparisonCSV/Excel of all applications Export all application data with scores for offline review, committee meetings, or executive decision-making Support decision-making outside Lennd — print ranked lists for a selection committee meeting, share with stakeholders who don't use the platform
