Module 1: Portals — the personalized stakeholder micro-site
Portals are the container module — not a feature in themselves, but the surface through which stakeholders interact with every other module. A portal is a dynamically generated website unique to each group, assembled from whichever modules the admin has assigned to that group type. Think of it as each vendor/artist/sponsor getting their own branded event website.
Phase 1 — Portal architecture setup (admin, one-time per group type)
ElementsTasksObjectives
Group type selectionWhich stakeholder type is this portal for Select the group type to configure (e.g., Vendors). All groups of this type will inherit this portal layout. Navigate to Portals in left-hand admin navigation Define the portal template once at the type level — not per vendor. 47 vendors all get the same portal structure automatically
Intake form setupFirst thing stakeholders see Build the onboarding intake form: what fields appear when a new group first opens their portal. Company name, primary contact, description, category — configurable per type Capture baseline group data on first visit — the intake form is the front door that populates the CRM before anything else happens
Module assignmentWhich modules appear in this portal Toggle on/off: credentials, catering, assets, guest lists, forms, documents, schedules, files, messages, event info, health checks. Each toggle adds a section to the portal Each group type sees only the modules relevant to their role — vendors get assets, artists get guest lists, volunteers get schedules
Item block assignmentWhat's requestable per module For each enabled module, assign or create item blocks: which credential types, meal types, asset types are available to request. Set quantities, limits, and pricing per block Fine-grained control over what each group type can request — vendors get vendor passes (not all-access), standard meals (not VIP dining)
Appearance customizationWelcome message and instructions Write a custom welcome message, add instructional text per section, list deadlines, include contact info for the admin. Customize what the stakeholder reads when they first land Make the portal feel intentional and guided — not a generic form dump, but a structured onboarding experience
Close datePortal submission deadline Set a date after which the portal locks for new submissions. Existing data remains viewable but not editable. Admin can override to reopen Enforce deadlines structurally — the system locks the door, not an email saying "please submit by Friday"
Phase 2 — Portal generation & distribution (admin → stakeholders)
ElementsTasksObjectives
Auto-generationPortal created per group When admin assigns forms and modules to a group type, portals auto-generate for each group of that type. No manual per-group setup needed. Each portal gets a unique URL Zero per-vendor portal creation effort — the system does it. Admin configures the template; the system stamps out 47 instances
Contact assignmentWho gets login access Add primary contact per group (auto from CRM if returning group). Grant login access checkbox per person. Additional contacts can be added with portal access Multiple people per vendor can access the same portal — the AV coordinator and the AV company owner can both log in and contribute
Invite emailBulk or individual send Send portal invite emails to all groups of a type in one click, or individually. Email contains unique portal link + welcome message. Can resend invitations for non-responders One-click distribution to dozens or hundreds of stakeholders — the portal link is the only thing they need to start interacting
Group-specific additionsOverride type-level config per group Add specific forms, document requests, or item blocks to an individual group's portal beyond the type-level defaults. E.g., headliner artist gets an additional hospitality form that opening acts don't Handle exceptions without creating a new group type — customize individual portals when the template doesn't cover edge cases
Phase 3 — Stakeholder portal experience (external users)
ElementsTasksObjectives
Portal home / dashboardLanding page per group Stakeholder logs in and sees: action items due, forms to complete, request statuses, upcoming deadlines, event info. All in one personalized page Immediate clarity on "what do I need to do?" — the portal answers this in 3 seconds without any email or phone call
Module navigationOnly assigned modules visible Navigation shows only enabled modules: Credentials, Catering, Assets, Forms, Files, Event Info, Messages — whatever the admin assigned. No locked or hidden items Structural simplicity — the stakeholder sees a focused, relevant interface, not a full platform with grayed-out features
Dynamic contentPortal updates as data changes Unlike a static form, the portal is a living workspace. Stakeholders return to check approval statuses, update personnel names, add new requests, view approved allocations Ongoing self-service relationship — not a one-time submission but a persistent communication and status surface
Completed vs. pending trackingVisual progress per section Each portal section shows completion: forms submitted vs. outstanding, credentials approved vs. pending, documents uploaded vs. requested Stakeholder self-monitors their own compliance — they know they're behind without the admin telling them
Real-time approval visibilityStatus without asking Every request (credentials, assets, catering, guest list) shows current status: submitted, pending review, approved, denied — with admin notes on denials Eliminate the #1 email from stakeholders: "What's the status of my request?" — the portal answers it live
Phase 4 — Admin monitoring & portal management
ElementsTasksObjectives
Onboarding completion dashboardCross-group progress Admin views completion % across all groups: who has finished their portal tasks, who is partial, who hasn't started. Filterable by group type, deadline status, module One screen answers "are we on track for onboarding?" — flag the 6 vendors who haven't started before it becomes a day-of crisis
Automated remindersSystem-sent deadline nudges Configure automatic email reminders for groups with incomplete portal tasks. Set frequency, message content, and which actions trigger the reminder The system chases — not the human. Admin sets it once; the platform nudges non-compliant groups automatically
Portal reopen / extendPost-deadline flexibility Reopen a closed portal for a specific group or extend the deadline for all groups. Handles late onboarding without rebuilding anything Accommodate real-world flexibility (vendors confirmed late, last-minute sponsor additions) without breaking the deadline-enforcement model
