Module: Badge printing — credential design, production & issuance
Badge printing is the physical output layer of the Credentials module. While Credentials handles the data (who is approved for what), Badge Printing handles the artifact (the physical badge, pass, or credential that a person wears, carries, or scans). It only becomes relevant after credentials are approved.
Phase 1 — Badge design & template setup (admin, pre-event)
ElementsTasksObjectives
Template per credential typeUnique design for each role Create distinct badge designs for each credential type: different background colors, layouts, and branding for Vendor, Media, VIP, Artist, Volunteer, Staff, All-Access. Each is visually distinguishable at a glance Security personnel can identify access level by badge appearance alone — a green vendor badge instantly reads different from a red all-access badge, no barcode scan needed for visual triage
Variable data fieldsWhat prints per person Configure which data fields print on each badge type. Options: full name, company/group name, title/role, photo, credential type label, event name, barcode/QR code, access zones. Different types can show different fields (e.g., "Volunteer Committee" on volunteer badges, "Company Name" on vendor badges) Each badge carries exactly the information relevant to that role — security sees what they need, the person sees their identity, and the barcode enables digital verification
Barcode / QR code placementScannable identifier on badge Place barcode or QR code in a consistent position on the badge. Code links to the person's credential record in Lennd. Configure code format to match ticketing partner requirements (Eventbrite, Intellitix) Every printed badge is a scannable credential — bridges the physical badge to the digital record for instant verification at any check-in point
Event brandingLogo, colors, sponsor marks Add event logo, sponsor logos, year/edition, and event-specific design elements. Ensure brand consistency across all badge types while maintaining type differentiation Badges double as brand assets — the credential is also a professional, branded artifact that reinforces the event's identity
Badge stock / format specificationPhysical medium selection Define badge format: laminated card, paper insert in lanyard pouch, adhesive label, wristband with printed insert. Specify dimensions, orientation (portrait/landscape), and material Match the physical format to the event's security and practical needs — a multi-day conference needs a durable laminated badge; a one-day festival can use paper in a pouch
Phase 2 — Pre-event batch printing (admin, days before event)
ElementsTasksObjectives
Print queue from approved credentialsBatch generation Generate print batch from all approved credentials. Filter by credential type, group, or alphabetical. System pulls name, company, photo, barcode from person records and merges with badge template Convert the approved credential list into a production-ready print queue with zero manual data entry — every approved person auto-populates a badge
Photo inclusionUploaded during credentialing If photos were collected during the credential request process (via portal form), they auto-populate onto the badge. Admin can review and crop/rotate photos before printing Photo badges increase security — gate staff can visually match face to badge, deterring credential sharing without needing to check ID
Batch print executionMulti-badge print run Send batch to connected printer. Print hundreds of badges in sequence. System tracks which badges have been printed and which remain. Pauses for stock changes if needed Handle the heavy production lift before event day — 500 badges printed overnight, sorted, and ready for will-call by morning
Print verificationQuality check on output Review printed badges for quality: text legibility, barcode scannability, photo alignment, color accuracy. Reprint any failed badges. Mark batch as verified Catch print errors before they reach the person — a smudged barcode or wrong name is fixable now, not at the gate with 200 people in line
Sorting and packagingOrganized for will-call pickup Sort printed badges by pickup method: alphabetical for will-call, grouped by company for vendor check-in, bundled by credential type for VIP desk. Package with lanyards/pouches Smooth pickup experience — when a vendor arrives at will-call, their 12 badges are already bundled together, not scattered alphabetically across 500 badges
Phase 3 — Onsite printing (event day, real-time)
ElementsTasksObjectives
On-demand single badge printingWalk-ins and late approvals When a new credential is approved day-of (late addition, walk-in registration, day-of hire), staff prints a single badge immediately from the Lennd interface at the registration desk Handle the inevitable last-minute additions without delay — the person walks up, gets approved, and has a badge in hand within minutes
Onsite photo captureTake photo at registration desk For walk-ins or people who didn't submit a photo pre-event, take their photo on the spot using a tablet/webcam. Photo attaches to their person record and prints on the badge Every badge has a photo regardless of when the person was added — maintains consistent security standard between pre-registered and day-of attendees
Badge reprintReplacement for lost/damaged Look up person by name or barcode scan. Reprint their badge with all original data intact. Optionally void the old barcode and issue a new one to prevent duplicate badges in circulation Lost badge doesn't mean lost access — quick reprint maintains security (void old barcode) while keeping the person productive
Credential correctionFix errors at the desk Person arrives and badge has wrong name spelling, wrong credential type, or wrong company. Staff corrects data in Lennd, reprints immediately. Correction logged in audit trail Fix human errors in real time without sending the person away or making them wait — correct it, reprint it, move on
Multiple print stationsDistributed printing across venue Set up print stations at multiple locations: main registration, VIP entrance, backstage, production office. Each station connected to the same Lennd data, any can print any badge Decentralize badge production to reduce bottlenecks — a vendor arriving at the backstage gate doesn't need to walk to the main registration to get a reprint
Phase 4 — Tracking & post-event (admin)
ElementsTasksObjectives
Issuance trackingPrinted vs. picked up vs. outstanding Track per-badge status: printed (ready at will-call), picked up (person collected it), not printed (approved but badge not yet produced), reprinted (replacement issued). Dashboard shows gaps Know at any moment: how many badges are printed, how many are picked up, how many are still waiting — operational pulse check during event
Pickup confirmationWill-call distribution log When a person collects their badge at will-call, staff marks it as "picked up" in the system. Optionally scans the badge to confirm. Timestamp recorded Close the chain from approval → print → pickup → check-in. The admin can track the full credential lifecycle from request to gate entry
Print volume reportTotal badges produced Report total badges printed by type, station, day. Compare against approved credential count. Identify overprints (reprints, corrections) and underprints (no-shows never picked up) Procurement data for next year — if 300 badges were printed but only 240 picked up, order less badge stock next time
Void / deactivate badgesSecurity revocation If a credential is revoked (person fired, vendor removed, security incident), admin voids the badge in the system. Voided barcode returns "denied" on scan even if the physical badge still exists Revocation without retrieval — you don't need to physically collect the badge, just deactivate it digitally and it becomes a useless piece of plastic at any scan point
How these three modules interconnect: Payments processes the financial transaction for a paid credential. Badge Printing produces the physical artifact for an approved credential. Check-in App verifies the scanned artifact against the digital record. The lifecycle is: request → approve → pay (if priced) → print → distribute → scan → verify. All three modules read from and write to the same person/credential record in the CRM — they're not separate systems, they're three views of the same data at different stages of the credential lifecycle.
