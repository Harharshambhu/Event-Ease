Module: Team management — internal user access & permissions
Team management controls who on the event organizer's internal staff can access what in Lennd. This is separate from the portal system (which controls external stakeholders). Think of it as: portals manage outward-facing access, team management manages inward-facing access. A credential manager shouldn't be able to change billing settings. A catering check-in operator doesn't need CRM access.
User invitation & onboarding
ElementsTasksObjectives
Invite via emailPrimary onboarding method Navigate to Team section. Click "Invite User." Enter first name, last name, and email. Assign a permission profile. System sends invitation email from noreply@lennd.com with subject "[inviter name] has invited you to join Lennd." User clicks "View My Events," creates a password, and is onboarded Get internal team members into the platform quickly with a guided onboarding path — they receive an email, set a password, and they're in
Invitation URL fallbackFor email delivery issues If invitation email doesn't arrive (spam filters, blocked domains), admin can copy the unique invitation URL from the Team panel and send it manually via any channel. Each URL is unique per user per event Guarantee onboarding even when email delivery fails — never blocked by IT email filters or spam quarantine
Resend invitationFor users who missed the email From Team panel, click the actions menu (three dots) on a pending user and select "Resend Invite." Triggers a fresh invitation email Handle the common "I didn't get the email" scenario without recreating the user — one-click retry
User typesTeam members vs. external contacts Users are internal team members who access the admin platform. They are distinct from CRM contacts (groups/people) who access portals. Typical users: artist liaisons, credential managers, media relations staff, volunteer coordinators, vendor managers, sponsorship managers, catering operators Maintain a clear separation between "people who run the event" (users) and "people who participate in the event" (contacts) — different access models for fundamentally different relationships
Permission profiles — module-level access control
ElementsTasksObjectives
Permission profile creationNamed access templates Create named profiles: "Full Admin," "Credential Manager," "Catering Operator," "Read-Only Reporter," "Volunteer Coordinator." Each profile is a set of module-level access toggles. Profiles are reusable across multiple users Define access once, apply to many — when 3 new credential reviewers join the team, assign the same profile to all 3 instead of configuring each individually
Module access togglesWhich modules a profile can see For each permission profile, toggle access to: CRM, Credentials, Catering, Assets, Guest Lists, Forms, Portals, Schedules, Payments, Reports, Dashboards, Team, Settings. Each can be on or off. A "Credential Manager" profile might only have CRM + Credentials + Reports enabled Principle of least privilege — a catering check-in operator doesn't need to see vendor payment records or credential approval queues. Show only what's relevant to their job function
Approver privilegesPer-module approval authority Within each enabled module, toggle whether the user can approve/deny requests or only view them. A user with Credentials access but no approver privilege can see pending requests and generate reports, but cannot approve or deny Separate "visibility" from "authority" — the intern can monitor the credential dashboard and flag issues, but only the credential manager can approve passes
Admin auto-approvalAdmins bypass approval queues Users with full admin + approver privileges have their own orders auto-approved (e.g., when an admin creates a catering order, it's immediately approved). Non-admin users' requests go into the pending queue Speed up internal operations — the admin shouldn't have to approve their own actions. But non-admin staff submissions still go through review
Profile assignmentOne profile per user per event Each invited user is assigned one permission profile. Profile can be changed at any time from the Team panel by clicking the actions menu and selecting a different profile Flexible role changes — when the volunteer coordinator is promoted to operations manager, swap their profile instead of rebuilding their access from scratch
User lifecycle management
ElementsTasksObjectives
User search & filteringFind team members quickly Search team members by name or email in the Team panel. View their assigned permission profile, invitation status (pending / active), and last activity Quick access to user management — find the person, verify their access level, and adjust if needed
Deactivation / removalRevoking access Remove a user from the event team. Their access is immediately revoked. They can no longer log in to the event's admin dashboard. Historical actions (approvals they made, changes they logged) are preserved in audit records Clean offboarding — when a contractor's engagement ends or a staff member leaves, revoke access instantly without losing the record of what they did while active
Multi-event user managementSame person, different events A user can be invited to multiple events within the same organization, potentially with different permission profiles per event. The credential manager for Summit 2026 might only be a viewer on Festival 2026 Match access to responsibility — the same person may have different roles on different events. Their access should reflect the specific event, not a blanket organizational role
Scan station user accountsFor onsite check-in devices Create dedicated user accounts for each scan station (one per gate, one per catering tent). These accounts have only check-in app access. Recommend one user per station so scan data is attributed to the correct physical location Track which physical station processed which scan — essential for post-event analysis of gate throughput, catering station utilization, and operational troubleshooting
Unlimited admin usersNo per-user seat limit on admin platform Lennd does not charge per admin user. Add as many team members as needed — credential manager, 3 catering operators, 5 gate scan stations, volunteer coordinator, production manager — all at no extra seat cost Remove the economic friction of "should we give this person access?" — if they need it for the event, add them. No seat-count anxiety like Monday.com or Jira
